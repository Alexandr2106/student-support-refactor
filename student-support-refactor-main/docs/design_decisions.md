# Решения по проектированию (Практическая №2)

## Почему фабричный метод?
Фабричный метод применён для **централизованного выбора реализации источника данных** на основе конфигурации. Это позволяет:
- Изолировать логику создания от бизнес-правил.
- Легко добавлять новые типы источников (например, Kafka, REST API).
- Не засорять доменную логику условиями вида `if type == "db"`.

## Как организована подмена?
- Все зависимости (`IRequestSource`, `INotifier`, `IAuditLogger`) передаются через **конструктор**.
- В тестах используется **ручная заглушка `FakeRequestSource`** (можно заменить на `Mock`).
- Конфигурация (`config.yaml`) читается **только в фабрике**, а не в бизнес-логике.

## Как это улучшило тестируемость?
- Тесты больше **не зависят от файловой системы, сети или БД**.
- Можно моделировать любые сценарии: дубликат, ошибка сохранения, разные ID.
- Каждый компонент тестируется **изолированно**.

Теперь система соответствует принципам **SOLID**: инверсия зависимостей (DIP), единственная ответственность (SRP), открытость/закрытость (OCP).
